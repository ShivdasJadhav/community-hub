# Contributing to the Project

First of all, thank you for taking the time to contribute! ❤️

Whether you're fixing a typo, improving documentation, designing UI, reporting bugs, or implementing a new feature, every contribution is appreciated.

This project aims to provide a real-world, production-inspired codebase where developers can learn modern software engineering practices while contributing to open source.

---

# Code of Conduct

Please be respectful and welcoming to everyone.

We expect contributors to:

- Be respectful in discussions.
- Provide constructive feedback.
- Help maintain a positive learning environment.
- Respect different experience levels.

---

# Before You Start

Before opening a Pull Request:

- Check existing Issues and Pull Requests.
- Search before creating a new Issue.
- Discuss large changes before implementing them.
- Keep Pull Requests focused on a single change.

---

# Development Workflow

1. Fork the repository.
2. Clone your fork.
3. Create a new branch.
4. Make your changes.
5. Test your changes.
6. Commit with a meaningful message.
7. Push your branch.
8. Open a Pull Request.

---

# Branch Naming

Please use descriptive branch names.

Examples:

```
feature/community-events

feature/inventory-dashboard

fix/login-validation

fix/booking-overlap

docs/api-authentication

refactor/member-service

test/community-controller
```

---

# Commit Messages

Please write meaningful commit messages.

Examples:

```
feat(events): add RSVP support

fix(auth): validate refresh token

docs(readme): update setup guide

refactor(users): simplify service layer

test(inventory): add repository tests
```

---

# Pull Request Guidelines

A good Pull Request should:

- Have a clear title.
- Explain the problem.
- Explain the solution.
- Include screenshots for UI changes.
- Pass all tests.
- Keep changes focused.

Avoid mixing unrelated changes into a single Pull Request.

---

# Coding Standards

Please follow the project's coding conventions.

General principles:

- Write readable code.
- Prefer simplicity.
- Avoid premature optimization.
- Use meaningful variable names.
- Keep functions focused.
- Remove dead code.
- Keep files organized.

---

# Project Structure

Each feature should remain modular.

Example:

```
feature/
├── api/
├── domain/
├── services/
├── repository/
├── schemas/
├── models/
└── tests/
```

Avoid tightly coupling unrelated modules.

---

# Reporting Bugs

When reporting a bug, include:

- Steps to reproduce
- Expected behavior
- Actual behavior
- Screenshots (if applicable)
- Browser / OS
- Error logs

---

# Feature Requests

A good feature request includes:

- Problem statement
- Proposed solution
- Alternative approaches
- Additional context

---

# Good First Issues

If you're new to the project, look for issues labeled:

- `good first issue`
- `documentation`
- `frontend`
- `backend`
- `bug`
- `enhancement`

These are intended to help new contributors get started.

---

# Reviewing Contributions

Every Pull Request will be reviewed for:

- Code quality
- Maintainability
- Readability
- Project consistency
- Testing
- Documentation

Review feedback is part of the collaborative process. Don't be discouraged by requested changes—they help keep the project healthy.
